Troubleshooting

Common issues encountered while configuring, running, or contributing to the LG Collared Report Workflow, and how to resolve them. If your issue isn't listed here, open an issue on GitHub.

Configuration & Connections

Subject Group returns no data

The Subject Group Name field is case-sensitive and must match the name in EarthRanger exactly. Also check the group doesn't mix subtypes unexpectedly — a group containing non-lion subjects alongside collared lions may produce unexpected results.

EarthRanger connection fails or times out

Verify the server URL, username, and password configured in the Data Source Connection. Confirm you can log into that same EarthRanger instance directly in a browser with the same credentials. Credentials aren't validated at setup time, so an invalid password only surfaces once the workflow actually runs.

Study area layers (Conservancies / Group Ranch Boundaries) are missing from the maps

Unlike some other STE workflows, these layers are not downloaded from Dropbox — they're fetched live from your connected EarthRanger instance as spatial features of type Conservancies and Group Ranch Boundaries. If your EarthRanger instance doesn't have spatial features registered under those exact names, both layers will simply be absent from every map, with no separate error.

Basemap tiles don't load, or the background map is blank

The Basemap Layers URLs are user-editable. If you changed either the ESRI World Hillshade or World Street Map layer away from its default, confirm the new tile URL follows the standard {z}/{y}/{x} template and is reachable from the machine running the workflow.

Report has one section total instead of one per lion (or vice versa)

Check the Groupers selection — Subject Name is pre-selected by default, which produces one map/metrics/report section per individual lion. If you want aggregated output instead, change the grouper to Subject Sex or Subject Subtype, or remove it to combine everything into a single view.

Trajectories & Map Outputs

Home Range or Subject Tracks maps are empty or sparse for a subject

This is usually the Trajectory Filter discarding most of the subject's data. If a lion moves unusually slowly or quickly relative to the default 1–30 km/h bounds, or has irregular fix intervals relative to the 10 s–6 h duration bounds, loosen the thresholds to match its actual movement characteristics, then re-run.

Home Range map looks unusually smoothed or noisy

Adjust the Time Density Map settings — Shape Buffer Expansion Factor (default 1.3) controls how far density values spread across the grid; lower it for a tighter, less-smoothed home range, or raise it for smoother contours. Max Speed Factor (default 1.05) bounds the kernel spread relative to the fastest measured segment.

Map is zoomed too tight or too loose around the data

Adjust the Zoom to Envelope Expansion Factor under Time Density Map and Zoom to Envelope — 1.0 is a tight fit to the data extent; increase it (e.g. 1.2) to add padding around the map.

No outputs written / can't find results

All outputs are written to the directory specified by the ECOSCOPE_WORKFLOWS_RESULTS environment variable. Confirm it's set and points to a directory the workflow process can write to.

Contributing & CI

These apply if you're modifying the workflow itself (spec.yaml, test-cases.yaml, or the generated ecoscope-workflows-collared-report-workflow/ directory) and opening a pull request.

Generated files differ from committed files in ecoscope-workflows-collared-report-workflow

The generated workflow directory is out of sync with spec.yaml. Run the recompile script locally and commit the result:

pixi run --manifest-path pixi.toml --locked bash -c "./dev/recompile.sh --update"

Found wildcard version (*) in requirements or Found local file channel (file://) in requirements

In spec.yaml, every dependency under requirements must pin an exact version and use a remote conda channel — wildcard versions and local file:// channels are only valid for local development, not for CI using published packages.

VERSION.yaml (...) must be greater than main branch (...)

Bump MAJ/MIN/PATCH in ecoscope-workflows-collared-report-workflow/VERSION.yaml so it's strictly greater than the version on main before opening or updating a PR.

test-cases.yaml must contain at least one test case

Every workflow requires at least one top-level entry in test-cases.yaml with a params block. See the existing entries in that file for the expected format.

Workflow tests fail in CI but pass locally (or vice versa)

test.yml runs test cases against all of ubuntu-latest, macos-latest, and windows-latest. If a test only fails on CI, check whether it depends on EarthRanger credentials being available as secrets, on the Conservancies/Group Ranch Boundaries spatial features existing on the test EarthRanger instance, or on OS-specific path handling.
